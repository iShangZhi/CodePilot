# 机制 Skill 提炼提示词

## 产出说明

每次提炼产出 **两个 SKILL**，分别写入两个独立目录：

| SKILL | 目录 | 使用场景 | 核心内容 |
|-------|------|---------|---------|
| 部署 | `product/skills/{机制名称}部署/` | **从零搭建**：新模块第一次接入机制时，按顺序依次执行各机制的部署 SKILL，节节相扣完成框架骨架 | Maven 依赖、框架基类继承、框架注解、必填配置、验证清单 |
| 服务清单 | `product/skills/{机制名称}服务清单/` | **日常迭代**：业务功能开发过程中随时查阅，了解机制有什么能力、怎么调用、怎么扩展 | 外部可调用 API（框架标准+自定义）+ 调用示例、常用 /data/ 端点、业务扩展点（SPI）、Service public 方法 |

> 两个 SKILL 共用同一轮源码解析结果（第一阶段），分别在第二阶段生成。

### 关键边界原则

**部署 SKILL 只包含「如何接入框架」**：
- 框架提供的基类、接口、注解，以及如何继承/实现它们
- 框架自动暴露的标准端点（CrudController / CombineController / CategoryController 等 default 方法）
- 框架要求的必填配置、i18n key、Maven 依赖
- **不包含**业务自定义的 API 方法、Controller 端点、Service 业务方法

**服务清单 SKILL 回答「有什么、怎么用、怎么调」**：

服务清单面向日常业务迭代，开发者随时查阅。ref 文件不只是方法签名列表，必须同时提供调用方式（如何注入、如何构造参数）和典型业务场景的代码示例，让开发者拿来就能写代码。

具体内容范围：
- **业务扩展点**：业务模块可实现的 SPI 接口/监听器/回调；含接口签名、触发时机、默认行为、实现代码模板
- **外部调用 API**：机制 `-api` 模块暴露的 Service 接口，含框架标准方法和自定义扩展方法；含注入方式和典型调用示例
- **常用 /data/ 端点**：常用 HTTP 端点（框架标准常用端点 + 全部自定义端点），以业务代码实际调用频率为筛选标准；含完整权限注解写法
- **Service public 方法**：供其他 Service/Controller 跨模块调用的公共方法；含调用场景说明
- **不包含**：框架内部方法、极低频标准端点（init/meta 等纯框架使用的端点）

**两个 SKILL 均不标注 `@OperLog`**：
- `@OperLog` 属于业务个性化配置（记录哪些操作、日志级别），由开发者根据实际需求自行添加
- SKILL 中的代码模板和端点描述一律省略 `@OperLog`，避免生成代码时产生冗余或错误的日志配置

## 第一阶段：源码解析

> 先读机制源码建立能力边界，再用参考工程验证接入方式。两者缺一不可。

```
分析以下两个来源，提炼 {机制名称} 的可复用 Skill：

- 机制源码：{mechanism_source}/      ← 先读，确立机制能力边界
- 参考工程：{reference_project}/{分析子模块}/   ← 后读，验证真实接入方式

### Step A — 读取机制源码（明确机制能力边界）

读取 {mechanism_source}/，识别：

1. **部署侧（业务模块需要做的）**
   - 必须/可选继承的基类、实现的接口、使用的注解
   - 需要引入的 Maven 依赖（artifactId + 引入位置）
   - 必填/可选的配置 key

2. **服务侧（机制对外暴露的）**
   - API 接口：机制的 `-api` 模块对外暴露的 Service 接口、方法签名（含框架标准方法）
   - Controller 接口：机制注册的 HTTP 路由、请求/响应结构
   - 扩展点：业务模块可实现的 SPI 接口/监听器/回调（如有）

输出：

**能力边界表**
| 能力项 | 类/接口/注解 | 归属（部署侧/服务侧） | 部署阶段（必须/按需） |
|--------|------------|-------------------|------------------|
| ...    | ...        | ...               | ...              |

**Maven 依赖清单**
| artifactId | 引入模块 | 归属 |
|-----------|---------|------|
| ...       | ...     | ...  |

---

### Step B — 参考工程接入扫描

基于 Step A 确立的边界，在参考工程中定位接入信号：

1. **Maven 依赖**：确认 Step A 中各 artifactId 是否存在、在哪个子模块
2. **部署侧信号**：基类继承、注解使用、配置 key
3. **服务侧信号**：对机制 API 接口的调用、对机制 Controller 路由的使用
4. **扩展点信号**：参考工程中对 SPI 接口/监听器的实现（如有）
5. **配置文件**：module.properties、ApplicationResources*.properties 中机制相关 key

输出：
| 识别信号（文件:行） | 归属（部署侧/服务侧） | 所属能力 |
|-----------------|-------------------|---------|
| ...             | ...               | ...     |

---

### Step C — 逐能力拆解

对每个能力，按部署侧/服务侧分类，完整梳理涉及的所有文件和层次：

**部署侧能力：**
- Maven 依赖（groupId / artifactId / 是否继承 BOM）
- Entity 层：继承的基类、新增字段及注解
- VO 层：字段来源、与 Entity 映射关系
- 配置 / i18n：必填 key、i18n 条目（key 只写命名空间冒号后的部分）

**服务侧能力：**
- API 接口：接口全限定名、方法签名、参数/返回类型语义（含框架标准方法和自定义扩展方法）
- Controller 路由：HTTP 方法、路径规则、请求/响应结构；标注哪些端点为常用端点
- Service 方法：可直接调用的公共服务方法、入参出参说明
- 扩展点：SPI 接口/监听器全限定名、方法签名、触发时机、默认行为
- **跨模块 DTO/VO**：方法签名中涉及其他模块定义的 DTO/VO/常量类，必须标注其来源模块（artifactId）；这些类型将在 ref-00-bundle-spec.md 中显式声明

**业务特定代码标记**（需替换为占位符）：
- 列出所有出现模块名/业务字段/业务逻辑的位置，标注替换方式
```

## 第二阶段：生成两个 SKILL

> 基于第一阶段解析结果，分别生成部署 SKILL 和服务清单 SKILL。

### 2-A 生成 `{机制名称}部署` SKILL

```
基于第一阶段的**部署侧能力**分析，生成 {机制名称}部署 SKILL：

**边界约束**：只写「框架层」的内容——继承什么、实现什么、加什么注解、配什么依赖。
不写业务自定义的 API 方法、Controller 端点、Service 业务方法（这些属于服务清单 SKILL）。

对于 API 接口层：
- **写**：继承哪个框架接口（如 ICategoryApi<VO>），框架接口提供了哪些标准方法
- **不写**：业务自己在接口里扩展的方法（如 copy、loadParentNode 等）

对于 Controller 层：
- **写**：implements 哪个框架 Controller（如 CategoryController），框架 default 方法有哪些
- **不写**：业务自定义的端点（如 /copy、/findCategoryTree 等）

对于 Service 层：
- **写**：继承框架基类（如 AbstractServiceImpl），实现框架接口（如 ICategoryService），框架要求的方法（如 getEntityManager）
- **不写**：业务自定义的 public 方法（如 fillTreeNodeParent、copyByCategory 等）

1. **业务代码剥离**：模块名、业务字段、业务方法名替换为占位符（规范见文末）

2. **生成 SKILL.md**，按序包含：
   - frontmatter：name = `{机制名称}部署`；description 说明机制用途和初始化范围
   - **一、能力概览**：表格列出各能力项、核心类/注解、部署阶段（必须/按需）、触发条件；
     表格下方"**关键约束**"列出最易踩坑的注意事项（≤ 5 条）
   - **二、部署流程**：
     - 阶段一 · 初始化（必须）：Maven 依赖代码块 + 必须步骤表格
     - 阶段二 · 按需追加：可选能力表格（全为必须则省略）
   - **三、部署验证清单**：按阶段分组，每项含「检查项 / 验证方法 / 常见错误」三列
   - **四、代码注释规范**：各代码位置的注释要求 + 行内注释原则
   - **五、占位符说明**：所有占位符、含义、示例（统一用 KmNews 体系）

3. **生成 references/**：
   - **ref-00-bundle-spec.md**：所有 ref 涉及的跨模块包导入，按依赖层分组（**不含 Maven 片段**，Maven 依赖只在 SKILL.md 的部署流程中展示）
   - **有 Entity 的能力**：以 Entity 为切入点，在同一 ref 文件依次补全 VO → API（框架接口声明）→ Repository（框架扩展查询）→ Service（框架方法实现）→ Controller（框架 default 方法声明）；
     机制能力后缀用 `-spec`，引擎集成后缀用 `-integration`
   - **无 Entity 的独立能力**：每个能力单独一个 ref 文件

4. **输出目录**：
   product/skills/{机制名称}部署/
   ├── SKILL.md
   └── references/
       ├── ref-00-bundle-spec.md
       ├── ref-01-{Entity语义}-spec.md
       └── ref-0N-{独立能力}-spec.md
```

### 2-B 生成 `{机制名称}服务清单` SKILL

```
基于第一阶段的分析，生成 {机制名称}服务清单 SKILL，覆盖「机制对外提供的全部可用能力」：

**范围说明**：
- 业务扩展点：业务模块可实现的 SPI 接口/监听器/回调（含接口签名、触发时机、默认行为）
- 外部调用 API：机制 -api 模块全部外部可调用方法（框架标准 + 自定义扩展，不限于 custom 方法）
- 常用 /data/ 端点：常用 HTTP 端点（筛选依据：业务代码实际调用频率）；框架标准 CRUD 基础端点（init/add/get/update/delete/deleteAll）不列，其余按实际常用度取舍
- Service public 方法：供其他 Service/Controller 跨模块调用的公共方法

**不包含**：框架内部方法、极低频标准端点（init/meta 等纯框架使用的端点）

对于 API 接口层：
- **写**：所有外部可调用的方法——框架继承来的标准查询/操作方法 + 业务自定义扩展方法
- **不写**：框架内部调用、仅供框架自身使用的方法

对于 Controller 端点：
- **写**：list/search/tree 等常用查询端点（即使是框架标准端点）+ 全部自定义业务端点
- **不写**：init/add/get/update/delete/deleteAll（标准 CRUD 操作，部署 SKILL 已说明框架提供）

对于扩展点：
- **写**：SPI 接口名、方法签名、触发时机、不实现时的默认行为
- 如无扩展点则省略对应 ref 文件

对于 Service 层：
- **写**：public 方法中可被其他模块调用的（非框架接口标准实现，属于跨模块内部调用）
- **不写**：框架接口的标准方法实现

1. **业务代码剥离**：同上，替换为占位符

2. **生成 SKILL.md**（索引文件，只含概览，细节下沉到 ref）：
   - frontmatter：name = `{机制名称}服务清单`；description 说明机制对外暴露了哪些可用能力
   - **一、服务概览**：按实体域分组，列出可调用 API 数量、常用端点数量、扩展点数量，注明对应 ref 文件
   - **二、业务扩展点概览**：表格（扩展点接口 / 触发时机 / ref 链接）；如无扩展点则省略
   - **三、Service public 方法概览**：表格（Service 类 / 方法数量 / ref 链接），不列方法签名
   - **四、工具类概览**：表格（工具类 / 核心方法 / ref 链接）；如无工具类则省略
   - **五、权限角色与校验器速查**：核心角色常量表 + 校验器 ID 速查表（完整保留在 SKILL.md，作为调用方的快速参考）
   > SKILL.md 是导航层，不承载方法签名和代码示例

3. **生成 references/**（按实体域 + 横切关注点分文件）：

   **分拆原则**：
   - 以**实体域**为第一维度划分：每个核心实体对应一个 ref 文件，同时包含该实体的**外部可调用 API 方法**（框架标准+自定义）和**常用 /data/ 端点**
   - **Service public 方法** 独立一个 ref 文件
   - **业务扩展点** 独立一个 ref 文件（如有）
   - **工具类** 独立一个 ref 文件（如有）
   - **ref-00-bundle-spec.md**（必须）：调用方所需的跨模块 DTO/VO/枚举类型声明（**不含 Maven 依赖**，Maven 依赖在部署 SKILL 的 SKILL.md 中已说明）

   **ref 文件内容规范**（每个 ref 文件必须包含签名 + 调用示例，缺一不可）：

   - **API 方法**：
     - 方法签名表：接口全限定名 + 方法签名 + 参数语义 + 返回语义；标注「框架标准」或「自定义扩展」
     - 注入方式：`@Autowired` / `@Resource` 注入接口的代码片段
     - 典型调用示例：按业务场景组织（如"查询用户可见列表"、"校验权限后执行操作"），注释说明"为什么这样调"

   - **端点**：
     - 端点表：HTTP 方法 + 路径 + 权限注解（完整注解写法）+ 说明；特殊处理须用"注意"块标注
     - 调用示例：前端/跨服务调用时的请求体结构示例（JSON），说明必填字段和业务含义

   - **扩展点**：
     - 接口名 + 方法签名 + 触发时机 + 不实现时的默认行为
     - 完整实现代码模板（含 `@Component` / `@Service` 注解、方法体骨架），开发者可直接复制修改

   - **Service public 方法**：
     - 方法签名表：类名 + 方法签名 + 参数/返回语义 + 适用场景
     - 典型调用示例：说明在哪种业务逻辑中需要跨模块调用此方法

4. **输出目录**：
   product/skills/{机制名称}服务清单/
   ├── SKILL.md                                  ← 索引：概览表 + 权限速查
   └── references/
       ├── ref-00-bundle-spec.md                 ← 跨模块类型声明（不含 Maven 依赖）
       ├── ref-01-{实体A}-guide.md               ← 实体A：外部可调用 API + 常用端点 + 调用示例
       ├── ref-02-{实体B}-guide.md               ← 实体B：外部可调用 API + 常用端点 + 调用示例
       ├── ref-03-service-guide.md               ← 全部 Service public 方法 + 调用示例
       ├── ref-04-extension-guide.md             ← 业务扩展点 + 实现模板（无则省略）
       └── ref-05-utils-guide.md                 ← 工具类 + 使用示例（无则省略）
```

## ref 文件内部结构规范

两类 SKILL 的 ref 文件侧重点不同：

### 部署 SKILL 的 ref（侧重"怎么接入框架"）

```
# ref-XX — {能力/接口名称}
> 一句话定位：这个能力解决什么问题，何时需要接入

**核心规则**
- 规则1（决策型：选什么、为什么、限制条件）
- ...（3-5 条，不超过 6 条）

**{步骤/层次标题}**（如"Entity 层"、"Service 层"）
```java
// 代码骨架（注释说明"为什么这样继承/实现"）
```

**易踩坑**
- 只写与直觉相悖、容易混淆的点，不超过 4 条
```

### 服务清单 SKILL 的 ref（侧重"怎么用、怎么调"）

```
# ref-XX — {实体/能力名称}
> 一句话定位：这个能力/接口解决什么业务问题，典型使用场景

**核心规则**
- 规则1（调用约束：参数要求、权限前提、异步/同步区别等）
- ...（3-5 条，不超过 6 条）

## 占位符规范

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `{Module}` | 模块名（大驼峰） | `KmNews` |
| `{module}` | 模块名（小驼峰） | `kmNews` |
| `{模块Key}` | 模块标识（kebab-case） | `km-news` |
| `{包路径}` | Java 包路径 | `com.landray.km.news` |
| `{ROLE}` | 角色前缀（全大写） | `KMNEWS` |
| `{Entity}` | 实体名（大驼峰） | `KmNewsMain` |
| `{entity}` | 实体名（小驼峰） | `kmNewsMain` |
| `{表名}` | 数据库表名 | `KM_NEWS_MAIN` |
