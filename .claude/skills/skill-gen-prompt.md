# 机制 Skill 提炼提示词

## 产出说明

每次提炼产出 **两个 SKILL**，分别写入两个独立目录：

| SKILL | 目录 | 面向 | 内容 |
|-------|------|------|------|
| 部署 | `product/skills/{机制名称}部署/` | 集成机制的开发者 | Maven 依赖、基类继承、注解、配置 |
| 服务清单 | `product/skills/{机制名称}服务清单/` | 调用机制能力的业务代码 | 机制暴露的 API 接口、Controller 路由、Service 方法签名 |

> 两个 SKILL 共用同一轮源码解析结果（第一阶段），分别在第二阶段生成。

---

## 第一阶段：源码解析

> 先读机制源码建立能力边界，再用参考工程验证接入方式。两者缺一不可。

```
分析以下两个来源，提炼 {机制名称} 的可复用 Skill：

- 机制源码：{mechanism_source}/      ← 先读，确立机制能力边界
- 参考工程：{reference_project}/{分析子模块}/   ← 后读，验证真实接入方式

### Step A — 读取机制源码（明确机制能力边界）

读取 {mechanism_source}/，识别：

1. **部署侧（业务模块需要做的）**
   - 必须/可选继承的基类、实现的接口、使用的注解
   - 需要引入的 Maven 依赖（artifactId + 引入位置）
   - 必填/可选的配置 key

2. **服务侧（机制对外暴露的）**
   - API 接口：机制的 `-api` 模块对外暴露的 Service 接口、方法签名
   - Controller 接口：机制注册的 HTTP 路由、请求/响应结构
   - 扩展点：业务模块可实现的 SPI 接口（如有）

输出：

**能力边界表**
| 能力项 | 类/接口/注解 | 归属（部署侧/服务侧） | 部署阶段（必须/按需） |
|--------|------------|-------------------|------------------|
| ...    | ...        | ...               | ...              |

**Maven 依赖清单**
| artifactId | 引入模块 | 归属 |
|-----------|---------|------|
| ...       | ...     | ...  |

---

### Step B — 参考工程接入扫描

基于 Step A 确立的边界，在参考工程中定位接入信号：

1. **Maven 依赖**：确认 Step A 中各 artifactId 是否存在、在哪个子模块
2. **部署侧信号**：基类继承、注解使用、配置 key
3. **服务侧信号**：对机制 API 接口的调用、对机制 Controller 路由的使用
4. **配置文件**：module.properties、ApplicationResources*.properties 中机制相关 key

输出：
| 识别信号（文件:行） | 归属（部署侧/服务侧） | 所属能力 |
|-----------------|-------------------|---------|
| ...             | ...               | ...     |

---

### Step C — 逐能力拆解

对每个能力，按部署侧/服务侧分类，完整梳理涉及的所有文件和层次：

**部署侧能力：**
- Maven 依赖（groupId / artifactId / 是否继承 BOM）
- Entity 层：继承的基类、新增字段及注解
- VO 层：字段来源、与 Entity 映射关系
- 配置 / i18n：必填 key、i18n 条目（key 只写命名空间冒号后的部分）

**服务侧能力：**
- API 接口：接口全限定名、方法签名、参数/返回类型语义
- Controller 路由：HTTP 方法、路径规则、请求/响应结构
- Service 方法：可直接调用的服务方法、入参出参说明
- **跨模块 DTO/VO**：方法签名中涉及其他模块定义的 DTO/VO/常量类，必须标注其来源模块（artifactId）；这些类型将在 ref-00-bundle-spec.md 中显式声明

**业务特定代码标记**（需替换为占位符）：
- 列出所有出现模块名/业务字段/业务逻辑的位置，标注替换方式
```

## 第二阶段：生成两个 SKILL

> 基于第一阶段解析结果，分别生成部署 SKILL 和服务清单 SKILL。

### 2-A 生成 `{机制名称}部署` SKILL

```
基于第一阶段的**部署侧能力**分析，生成 {机制名称}部署 SKILL：

1. **业务代码剥离**：模块名、业务字段、业务方法名替换为占位符（规范见文末）

2. **生成 SKILL.md**，按序包含：
   - frontmatter：name = `{机制名称}部署`；description 说明机制用途和初始化范围
   - **一、能力概览**：表格列出各能力项、核心类/注解、部署阶段（必须/按需）、触发条件；
     表格下方"**关键约束**"列出最易踩坑的注意事项（≤ 5 条）
   - **二、部署流程**：
     - 阶段一 · 初始化（必须）：Maven 依赖代码块 + 必须步骤表格
     - 阶段二 · 按需追加：可选能力表格（全为必须则省略）
   - **三、部署验证清单**：按阶段分组，必须项标"必查"，可选项标"用到时查"
   - **四、代码注释规范**：各代码位置的注释要求 + 行内注释原则
   - **五、占位符说明**：所有占位符、含义、示例（统一用 KmNews 体系）

3. **生成 references/**：
   - **ref-00-bundle-spec.md**：所有 ref 涉及的跨模块包导入，按依赖层分组，附完整 Maven 片段
   - **有 Entity 的能力**：以 Entity 为切入点，在同一 ref 文件依次补全 VO → API → Repository → Service → Controller；
     机制能力后缀用 `-spec`，引擎集成后缀用 `-integration`
   - **无 Entity 的独立能力**：每个能力单独一个 ref 文件

4. **输出目录**：
   product/skills/{机制名称}部署/
   ├── SKILL.md
   └── references/
       ├── ref-00-bundle-spec.md
       ├── ref-01-{Entity语义}-spec.md
       └── ref-0N-{独立能力}-spec.md
```

---

### 2-B 生成 `{机制名称}服务清单` SKILL

```
基于第一阶段的**服务侧能力**分析，生成 {机制名称}服务清单 SKILL：

1. **业务代码剥离**：同上，替换为占位符

2. **生成 SKILL.md**，按序包含：
   - frontmatter：name = `{机制名称}服务清单`；description 说明机制提供哪些业务能力
   - **零、Maven 依赖**：调用方需要引入的依赖声明（groupId / artifactId / 引入模块 / 是否继承 BOM）
   - **一、服务概览**：表格列出所有 API 接口 / Controller 路由，标注用途和调用时机
   - **二、API 接口清单**：按接口分组，每个接口列出方法签名、参数语义、返回值、使用场景
   - **三、Controller 路由清单**：按功能分组，每条路由列出 HTTP 方法 + 路径 + 请求/响应结构
   - **四、使用约束**：调用前提条件、权限要求、事务边界说明

3. **生成 references/**：
   - **ref-00-bundle-spec.md**：调用方代码中涉及的所有包导入，按依赖层分组，附调用方所需的完整 Maven 依赖片段；
     所有后续 ref 文件的代码块只保留本类自模块的导入，跨模块导入统一在此声明。
     **重点**：方法签名中用到的来自其他模块的 DTO/VO/常量（如请求体、响应体、枚举），必须在此文件中：
     1. 声明其来源 Maven 依赖（groupId / artifactId）
     2. 列出完整类名及包路径
     3. 说明该类型属于哪个机制模块，避免调用方错误自行定义
   - 每个主要接口或路由分组对应一个 ref 文件，包含完整方法签名和调用示例代码

4. **输出目录**：
   product/skills/{机制名称}服务清单/
   ├── SKILL.md
   └── references/
       ├── ref-00-bundle-spec.md
       ├── ref-01-{接口组语义}-api.md
       └── ref-0N-{路由组语义}-controller.md
```

## 第三阶段：参考工程二次验证

> 用参考工程的实际代码校验两个 SKILL 的准确性，发现偏差立即修正。

```
对照参考工程 {reference_project}/{分析子模块}/ 的实际代码，
分别检查部署 SKILL 和服务清单 SKILL 是否存在以下偏差：

**部署 SKILL 校验点：**
1. 结构偏差：文件组织方式与参考工程不一致
2. 内置 API 遗漏：平台内置 id/实现未提及或描述错误
3. 注解用法偏差：注解属性填写方式与实际不符
4. 复合场景缺失：参考工程有更复杂的组合用法未覆盖
5. 条件性规则过于绝对：写了"必须"但实际有例外
6. Maven 依赖描述错误：artifactId 错误、scope 错误、遗漏

**服务清单 SKILL 校验点：**
1. 接口遗漏：参考工程实际调用了但 SKILL 未收录的 API
2. 方法签名错误：参数类型、返回类型与实际不符
3. 路由路径错误：HTTP 方法或路径规则与实际不符
4. 使用约束遗漏：实际有权限/事务要求但未说明

对每处偏差输出：
| SKILL | 偏差类型 | 原描述 | 修正后描述 |
|-------|---------|-------|---------|
| 部署/服务清单 | ... | ... | ... |

修正完成后更新受影响的 SKILL.md 和 ref 文件。
```

## ref 文件内部结构规范

```
# ref-XX — {能力/接口名称}
> 一句话定位：这个能力/接口解决什么问题、适用什么场景

**核心规则**
- 规则1（决策型：选什么、为什么、限制条件）
- 规则2
- ...（3-5 条，不超过 6 条；代码里已自说明的事实不写）

**{意图标题}**（直接描述用途，不用"场景一/二/三"编号）
```java
// 代码（注释说明"为什么"而非"做什么"）
```

**易踩坑**
- 只写与直觉相悖、容易混淆的点
- 不重复"核心规则"中已有内容
- 不超过 4 条
```

**格式要求**：`核心规则` 用 bullet list，每条以关键词加粗开头；代码具有通用性，不含业务字段。

## 占位符规范

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `{Module}` | 模块名（大驼峰） | `KmNews` |
| `{module}` | 模块名（小驼峰） | `kmNews` |
| `{模块Key}` | 模块标识（kebab-case） | `km-news` |
| `{包路径}` | Java 包路径 | `com.landray.km.news` |
| `{ROLE}` | 角色前缀（全大写） | `KMNEWS` |
| `{Entity}` | 实体名（大驼峰） | `KmNewsMain` |
| `{entity}` | 实体名（小驼峰） | `kmNewsMain` |
| `{表名}` | 数据库表名 | `KM_NEWS_MAIN` |
