# 机制 Skill 提炼提示词（源码逆向版）

## 适用场景

有一个已完整部署机制的案例工程（或模块），需要通过阅读其源码，
逆向解析各机制的接入方式，提炼为可复用 Skill，以便在其他模块中复现相同部署。

---

## 第一阶段：源码解析（识别机制 + 拆解接入方式）

> **两个输入来源，使用顺序固定：先读机制源码建立能力边界，再用案例工程验证接入方式。**
> 两者缺一不可：只读机制源码会遗漏真实接入细节；只读案例工程会误将业务代码纳入 Skill。

```
我需要你分析以下两个来源，提炼 {机制名称} 的可复用部署 Skill：

- 机制源码：@{mechanism_source}/{机制模块}/      ← 先读，确立机制能力边界
- 案例工程：@{reference_project}/{分析子模块}/   ← 后读，验证真实接入方式

请按以下步骤执行（需要的权限直接获取）：

### Step A — 读取机制源码（明确机制能力边界）

先读 `{mechanism_source}/` 的源码，明确该机制对外提供的完整能力：

1. **API 层**：读取机制的 `-api` 模块，提取所有对外接口、基类、注解定义
2. **核心扩展点**：归纳业务模块必须/可选继承或实现的类、接口、注解
3. **Maven 依赖**：记录业务模块需要引入的 artifactId（通常为 `-api` 模块）
4. **配置约定**：读取机制默认配置项、必填 key、可选 key（如有）

输出格式：

**能力边界表**
| 能力项 | 类/接口/注解 | 部署阶段 | 说明 |
|--------|------------|---------|------|
| ...    | ...        | 初始化必须 / 按需追加 | ...  |

**Maven 依赖清单**
| artifactId | 引入模块（api/core/client） | 说明 |
|-----------|--------------------------|------|
| ...       | ...                      | ...  |

> 此步骤确立机制能力边界——案例工程中凡是涉及这些类/接口/注解的代码，均属于机制接入代码，其余为业务代码。

---

### Step B — 案例工程接入扫描（确认信号）

基于 Step A 确立的机制边界，在案例工程中快速定位接入信号：

1. **Maven 依赖**：读取各模块 pom.xml，确认 Step A 中的 artifactId 是否存在
2. **基类 / 接口继承**：搜索 Entity、Service、API 层中继承或实现的机制框架类
3. **注解信号**：搜索 Step A 识别出的机制注解
4. **配置文件**：读取 module.properties、ApplicationResources*.properties，提取机制相关 key

输出格式：
| 识别信号（文件:行） | 所属能力 | 涉及模块（api/core/client） |
|-----------------|---------|--------------------------|
| ...             | ...     | ...                      |

---

### Step C — 逐能力接入拆解

对 Step A 识别出的每个能力，结合 Step B 的案例工程接入代码，完整梳理涉及的所有文件和层次：

**对每个能力输出以下结构：**

#### {能力名称} 接入清单

**Maven 依赖**
- 列出完整 `groupId` / `artifactId` / 是否需要显式 `version`（version 继承 BOM 时注明"继承 BOM，无需声明"）

**Entity 层**（如有）
- 文件路径、继承的基类、新增的字段及其注解、原因说明

**VO 层**（如有）
- 文件路径、字段定义来源、与 Entity 的映射关系

**API 层**（如有）
- 文件路径、实现/继承的机制接口、新增的方法签名

**Service 层**（如有）
- 文件路径、调用了哪些机制提供的服务方法、事务边界

**Controller 层**（如有）
- 文件路径、使用的校验注解、路由定义

**配置 / i18n**（如有）
- `module.properties` 新增的 key
- `ApplicationResources.properties` 新增的 i18n 条目：记录完整 key 名称、中文值、用途说明
  - 注意：`messageKey` / `desc` 中冒号前的模块命名空间（`{模块Key}:`）由平台自动剥离，properties key 只写冒号后的部分

**识别到的业务特定代码**（后续需替换为占位符）
- 列出所有出现 {模块名}/业务字段/业务逻辑 的位置，标注替换方式
```

## 第二阶段：Skill 生成（基于解析结果提炼通用 Skill）

> 在完成第一阶段解析后，将以下提示词附在解析结果后继续执行。

```
基于上方对 {机制名称} 的机制接入分析，提炼一个可复用的部署 Skill，
要求如下（需要的权限直接获取）：

1. **业务代码剥离**
   - 所有业务特定内容（模块名、业务字段、业务方法名）统一替换为通用占位符
   - 占位符规范见下方"占位符说明"一节
   - 保留机制接入的结构骨架：注解、基类继承、接口实现、配置 key 命名规律

2. **生成 SKILL.md**，结构如下（按序包含以下各节）：
   - **frontmatter**：只含 name 和 description 两个字段；
     name 与目录名一致（如：MK分类模板机制部署，不加引号）；
     description 说明机制用途，若有按需追加能力，注明"初始化仅需 XX；其余能力按业务场景追加"
   - **一、能力概览**：表格列出各能力项、核心类/注解、**部署阶段**（初始化必须 / 按需追加）、触发条件；
     表格下方以"**关键约束**"列出最容易踩坑的注意事项（不超过 5 条）
   - **二、部署流程**：分两个 Phase：
     - **阶段一 · 初始化（必须，每个模块只做一次）**：Maven 依赖代码块 + 必须步骤表格（步骤 0 = Maven，步骤 1 = 最小必须接入）
     - **阶段二 · 按需追加（由业务场景触发）**：以表格列出各可选能力、参考 ref 文件、何时追加；若所有能力都必须则省略 阶段二
   - **四、部署验证清单**：按 Phase 分组，阶段一 标"必查"，阶段二 各能力独立一组标"用到时查"
   - **五、代码注释规范**：表格列出各代码位置及注释要求；补充行内注释原则
   - **六、占位符说明**：表格列出所有占位符、含义及示例值（示例统一使用 `KmNews` 体系）

3. **生成 references/ 目录**，遵循以下规则：

   - **首先生成 ref-00-bundle-spec.md**：
     汇总该 Skill 所有 ref 文件涉及的跨模块包导入，
     按依赖层分组（JDK/标准库、Lombok、Hutool、Spring、Swagger、框架基础层、框架业务层等），
     每组用三级标题区分，末尾附完整 Maven 依赖片段；
     所有 ref 文件均须遵守此文件，ref 内代码块只保留本类自模块的导入

   - **其余 ref 文件按"有无 Entity"拆分**：
     - **有 Entity**：以 Entity 为切入点，在该文件中依次补全
       VO → API → Repository → Service → Controller 各层代码；
       机制能力后缀用 `-spec`，引擎集成后缀用 `-integration`
     - **无 Entity（独立能力）**：每个独立能力单独一个 ref 文件，
       如：角色定义、校验器使用、自定义校验器各自独立成 ref；
       不要把多个无关联能力塞进同一个 ref 文件

   - **每个 ref 文件内部结构**（严格遵守，不得混用旧格式）：

     ```
     # ref-XX — {能力名称}
     > 一句话定位：这个能力解决什么问题、适用什么场景

     **核心规则**
     - 规则1（决策型：选什么、为什么、限制条件）
     - 规则2
     - ...（3-5 条，不超过 6 条；代码里已自说明的事实不写在这里）

     **{意图标题}**（直接描述用途，如"后台管理接口"，不用"场景一/二/三"编号）
     ```java
     // 代码
     ```

     **{意图标题二}**（有多个场景时继续）
     ```java
     // 代码
     ```

     **{意图标题}**（配置 / i18n 场景，使用 properties 代码块）
     ```properties
     # 注释说明用途
     key.name=中文值
     key.desc=中文描述
     ```

     **易踩坑**
     - 只写与直觉相悖、容易混淆的点（如两个相似 API 的细微区别）
     - 不重复"核心规则"中已有的内容
     - 不超过 4 条
     ```

     **格式要求**：
     - `核心规则` 用 bullet list，不用表格；每条以关键词加粗开头（如 `- **文件位置**：...`）
     - `易踩坑` 替代旧的 `注意事项`；若与 `核心规则` 有重复，保留规则，删掉坑
     - 代码注释说明"为什么"而非"做什么"；行内注释标注每个关键参数的语义
     - 代码具有通用性，不含任何案例工程的业务字段

4. **输出目录结构**：
{机制名称}部署/
  ├── SKILL.md
  └── references/
      ├── ref-00-bundle-spec.md
      ├── ref-01-{Entity语义}-spec.md        ← 机制能力型（Entity + 全栈）
      ├── ref-02-{Entity语义}-integration.md ← 引擎集成型（Entity + 全栈）
      └── ref-0N-{能力语义}-spec.md          ← 无 Entity 的独立能力（每个能力单独一文件）
```

## 第三阶段：参考工程二次验证（必做）

> 纯靠机制源码提炼容易遗漏真实接入细节，必须用参考工程的实际代码校验 Skill 准确性。

```
请对照参考工程 @{reference_project}/{分析子模块}/ 的实际代码，
逐项检查上方生成的 Skill 是否存在以下类型的偏差，发现偏差立即修正：

1. **结构偏差**：类/文件的组织方式与参考工程不一致（如角色常量是 interface 还是 class）
2. **内置 API 遗漏**：平台提供了内置 id/实现（如 roleValidator），Skill 未提及或描述错误
3. **注解用法偏差**：注解属性的填写方式与实际用法不符（如 @ValidateParam 是否需要 name）
4. **复合场景缺失**：参考工程有更复杂的组合用法（如 @CompsiteValidators），Skill 未覆盖
5. **条件性规则过于绝对**：Skill 写了"必须"但实际有例外（如 @Component 的条件性）
6. **Maven 依赖描述错误**：artifactId 错误、scope 错误、或遗漏

对每处偏差，输出：
| 偏差类型 | 原描述 | 修正后描述 |
|---------|-------|---------|
| ...     | ...   | ...     |

修正完成后更新所有受影响的 SKILL.md 和 ref 文件。
```

## 占位符规范

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `{Module}` | 模块名（大驼峰） | `KmNews` |
| `{module}` | 模块名（小驼峰） | `kmNews` |
| `{模块Key}` | 模块标识（kebab-case） | `km-news` |
| `{包路径}` | Java 包路径 | `com.landray.km.news` |
| `{ROLE}` | 角色前缀（全大写，与模块 Key 对应） | `KMNEWS` |
| `{Entity}` | 实体名（大驼峰） | `KmNewsMain` |
| `{entity}` | 实体名（小驼峰） | `kmNewsMain` |
| `{表名}` | 数据库表名 | `KM_NEWS_MAIN` |
